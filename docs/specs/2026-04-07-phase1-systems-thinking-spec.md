## 1. Executive Summary

Phase 1 should no longer teach a fake linear method for wicked problems.

This version uses **backward design**:

1. define what teams must be able to do by the end
2. define what evidence will prove they can do it
3. build only the activities required to produce that evidence

The phase is for **871 high school participants**. They do not need more busywork. They need a strong learning arc that helps them understand a real-world system, gather evidence from real people, identify leverage points, and decide whether a problem is worth solving.

The core loop remains:

**see the system -> gather evidence -> map the system -> decide**

But the pedagogy now starts from the end state:

**Problem Proof Pack + Proceed / Pivot / Kill decision**

Everything in the phase must build toward that.

---

## 2. Why We Are Changing Direction

### What was wrong

The prior Phase 1 design had too many steps and taught the wrong mental model:

- too linear
- too much task completion
- too much duplication
- too much "content about process" instead of real sensemaking
- 5 Whys is too narrow for wicked problems

### Core decision

We are explicitly dropping **5 Whys as a central method**.

For wicked problems, participants need to see:

- actors
- incentives
- bottlenecks
- workarounds
- feedback loops
- unintended consequences
- leverage points

That means **systems thinking**, not root-cause theater.

---

## 3. Backward Design Principle

This phase should be designed in the same order the learning engine in this repo already implies:

1. **Desired results**  
   What should teams understand and be able to do?
2. **Acceptable evidence**  
   What artifacts or decisions will prove that understanding?
3. **Learning plan**  
   What activities are required to produce that evidence?

This matters because high school participants are especially vulnerable to confusing completion with understanding. The product should not reward them for "getting through the checklist." It should reward them for producing evidence that they understand a real problem inside a real system.

### Design rule

If an activity does not produce evidence needed for the final decision, it should be cut, merged, or moved to optional support content.

---

## 4. Stage 1: Desired Results

### Transfer goal

By the end of Phase 1, a team should be able to say:

> We understand the system around this problem, we have evidence from real people, we can explain what keeps the problem alive, and we know whether this is worth solving.

### End-of-phase capability

A successful team can:

- distinguish symptom from system
- explain how the problem currently works in the real world
- support claims with interview evidence instead of guesses
- identify at least one leverage point
- make a reasoned **Proceed / Pivot / Kill** decision

### Enduring understandings

Participants should leave Phase 1 understanding that:

- wicked problems are not solved by one clean root cause
- repeated behavior matters more than abstract opinions
- workarounds often reveal the real shape of the system
- a good problem statement is earned through evidence
- not every problem deserves a solution sprint

### Essential questions

The phase should keep returning to these questions:

1. What is happening in this system, not just at the surface?
2. Who is affected, and what are they optimizing for?
3. What patterns keep this problem alive?
4. What evidence proves this problem is real?
5. Is this a problem we should proceed with, pivot, or kill?

---

## 5. Stage 2: Acceptable Evidence

Phase 1 is successful only if teams leave with a **Problem Proof Pack** containing:

1. **Target user**
2. **System map**
3. **Interview evidence**
4. **Problem statement**
5. **Proceed / Pivot / Kill decision**

### Evidence standard

The final pack should prove four things:

#### A. Systems understanding

Teams can:

- identify the relevant actors
- describe the sequence of events in the system
- identify incentives and constraints
- explain at least one loop or recurring pattern
- identify a plausible leverage point

#### B. Evidence quality

Teams can:

- gather 5-10 real interviews
- capture behavior, not just opinions
- surface recurring pain points and workarounds
- identify surprising or disconfirming findings

#### C. Problem clarity

Teams can:

- define a specific target user
- write a sharp, evidence-backed problem statement
- explain why this problem matters in context

#### D. Decision readiness

Teams can:

- decide Proceed / Pivot / Kill
- justify the decision with evidence and system understanding
- name what would change their mind

### Minimum rubric for mentors and reviewers

The final submission should be judged on four dimensions:

1. **Reality**  
   Is this grounded in real observed behavior?
2. **Systems understanding**  
   Does the team understand what keeps the problem alive?
3. **Specificity**  
   Is the user, context, and problem statement sharp enough?
4. **Decision quality**  
   Is Proceed / Pivot / Kill justified by evidence?

### Assessment rule

Every core activity should produce one of the following:

- orientation toward the end goal
- raw evidence
- synthesized evidence
- decision-ready reflection
- a required final artifact

If it produces none of these, it is not core Phase 1 pedagogy.

## 6. Stage 3: Learning Plan

Phase 1 should have **7 core activities max**.

The sequence below is not just a product flow. It is an evidence-building arc.

### Activity 1 - What You'll Walk Away With

**Purpose**  
Set the promise and frame the phase correctly.

**Backward-design role**  
Show the end state first so students know what "good" looks like.

**Key message**  
You are not here to find an idea quickly. You are here to understand a system well enough to choose a meaningful problem.

**Format**  
Webtoon / comic / short intro visual.

**Learning effect**  
Prevents students from optimizing for novelty or speed.

**Output**  
Orientation only. No graded artifact.

---

### Activity 2 - See the System, Not Just the Symptom

**Purpose**  
Teach a simple systems-thinking lens before interviews begin.

**Backward-design role**  
Give students the seeing frame they will need in order to collect useful evidence later.

**Participants should learn to look for:**

- who is involved
- what happens first, next, and after that
- where the friction appears
- what workaround people use
- what incentives keep the problem alive
- what happens when someone tries to fix it
- where loops or repeated patterns exist

**Format**  
Interactive lesson, lightweight visual explainer, or guided tool walkthrough.

**Preferred tool**  
Loopy or a similarly simple causal loop mapping tool.

**Output**  
Rough System Map v1.

**Evidence value**  
Early diagnostic artifact. It should be rough, but it reveals whether teams are seeing the system at all.

---

### Activity 3 - Interview Real Humans

**Purpose**  
Gather evidence from reality, not opinions.

**Backward-design role**  
Generate the raw evidence required for both the final pack and the decision gate.

**Interview design principle**  
Do not teach 5 Whys. Teach students to trace the system.

**Question patterns should focus on:**

- what happened last time
- what they did step by step
- who else was involved
- what made it difficult
- what workaround they used
- what tradeoff they made
- what happened after the workaround
- what made the problem continue

**Avoid:**

- leading questions
- pitching the solution
- abstract opinions without behavior

**Output**  
5-10 real interviews.

**Evidence value**  
This is the core raw dataset for the whole phase.

---

### Activity 4 - Upload Evidence

**Purpose**  
Force teams to ground their claims in evidence that can be reviewed.

**Backward-design role**  
Turn invisible discovery work into inspectable evidence.

**Accepted inputs**

- interview notes
- clips
- audio
- observations
- screenshots of field notes

**Output**  
Evidence bundle.

**Evidence value**  
Creates the proof layer mentors can inspect later.

---

### Activity 5 - Map the System

**Purpose**  
Turn raw evidence into structure.

**Backward-design role**  
Move students from collecting anecdotes to explaining the system.

**Team should produce:**

- key actors
- recurring pain points
- existing workarounds
- incentives and constraints
- reinforcing or balancing loops
- possible leverage points

**Format**  
Team synthesis activity.

**Output**  
System Map v2.

**Evidence value**  
Primary synthesis artifact showing whether the team can interpret evidence, not just collect it.

---

### Activity 6 - Decision Gate: Proceed / Pivot / Kill

**Purpose**  
Make the team decide what the evidence actually means.

**Backward-design role**  
Assess decision quality before the final submission.

**Decision options**

- **Proceed** - strong enough evidence, clear leverage point
- **Pivot** - problem is real but framing or user focus is wrong
- **Kill** - weak evidence or low-value problem

**Required reflection**

Teams must answer:

1. What is the strongest evidence this problem is real?
2. What keeps this problem alive in the system?
3. Where is the leverage point?
4. Why are we proceeding, pivoting, or killing?
5. What evidence would make us change our mind?

**Output**  
Decision with reasoning.

**Evidence value**  
This is the clearest proof that students can move from research to judgment.

---

### Activity 7 - Submit Problem Proof Pack

**Purpose**  
Finalize the Phase 1 artifact.

**Backward-design role**  
Collect the full body of evidence that proves the phase goal was met.

**Required contents**

1. **Target User**
   - who specifically
   - context
   - why this group matters

2. **System Map**
   - actors
   - loops
   - bottlenecks
   - leverage point

3. **Evidence Summary**
   - 5-10 interviews
   - strongest patterns
   - surprising findings
   - workarounds observed

4. **Problem Statement**
   - specific
   - evidence-backed
   - grounded in system behavior

5. **Decision**
   - Proceed / Pivot / Kill
   - why

**Output**  
Problem Proof Pack.

**Evidence value**  
Summative artifact for product, mentor, and team review.

## 7. Pedagogical Guardrails

### What this phase should teach

- how to see a messy real-world problem as a system
- how to gather behavioral evidence
- how to synthesize evidence into structure
- how to make a reasoned decision under uncertainty

### What this phase should not teach

- that discovery is a neat checklist
- that one "root cause" explains a wicked problem
- that confidence matters more than evidence
- that a team must proceed just because they started

### Pedagogical stance

This phase should feel like:

- concrete
- evidence-first
- slightly demanding
- grounded in reality
- oriented toward judgment

It should not feel like:

- founder theater
- abstract innovation language
- fake rigor
- academic systems theory

---

## 8. Optional Support Workflow

### Mentor Review

Mentor review is optional as a learning activity and should be treated as support and evaluation workflow, not core pedagogy.

If included, it should come **after** the Problem Proof Pack so mentors are reacting to evidence and decision quality, not replacing the team's own judgment.

---

## 9. What We Are Cutting

The following should be removed from the core Phase 1 learning flow:

- Mascot 5-Whys
- 5 Whys as a named methodology
- separate persona-first exercise before evidence exists
- duplicate AI feedback steps
- NotebookLM guideline as a required activity
- trustful sources submission as a separate learning milestone
- mentor booking as a core learning activity

These may be:

- deleted
- merged into another activity
- moved to optional support content
- moved to later phases if still useful

---

## 10. New Learning Model

### Old model

Problem -> ask why -> find root cause -> define problem

### New model

Messy situation -> understand the system -> gather evidence -> find loops and leverage points -> define the right problem -> decide whether to continue

### Backward-designed model

Desired outcome -> required evidence -> learning experiences -> final decision

This is the core pedagogical shift.

---

## 11. Loopy / Systems Mapping Tool Requirements

We want a lightweight system-mapping tool experience.

### If using Loopy or equivalent, participants should be able to:

- name actors and elements
- connect cause/effect relationships
- show reinforcing loops
- show balancing loops
- mark bottlenecks
- identify possible intervention points

### Important constraint

Do **not** turn this into formal systems theory.

Students do not need academic vocabulary. They need practical prompts:

- What keeps this problem going?
- Who is stuck in it?
- What are they optimizing for?
- What workaround makes it worse?
- Where could a small intervention change a lot?

- teams upload real interview evidence
- teams identify a leverage point
